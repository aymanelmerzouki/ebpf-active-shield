# v0.3 : rendre le projet utilisable et propre

La v0.2 bloquait déjà, mais avec deux limites de "prototype" :

1. L'allowlist était **codée en dur** dans le code Rust du daemon.
2. Les événements remontaient via de simples **logs texte**.

La v0.3 corrige ces deux points :

1. **Allowlist chargée depuis un fichier de configuration** (`shield.toml`).
2. **Ring buffer** : un canal moderne pour remonter des événements structurés
   du noyau vers le daemon.

---

# 1. Pourquoi une config externe ?

Coder la liste des programmes autorisés en dur dans le code, c'est pratique pour
un prototype, mais inutilisable en vrai : il faudrait recompiler à chaque
changement.

En v0.3, on lit la liste depuis un fichier `shield.toml` :

```toml
mode = "log-only"   # ou "enforce"
allowed_callers = ["bash", "zsh", "systemd", "cargo", "git"]
```

Le daemon lit ce fichier au démarrage, remplit la map, et applique le mode.
Changer la politique = éditer le fichier, plus besoin de recompiler.

C'est la séparation **code / configuration** : un principe de base en ingénierie
logicielle, très regardé en entretien.

---

# 2. Le concept clé : le ring buffer

## Le problème des logs

En v0.2, le programme eBPF envoyait des messages avec `info!()`. C'est pratique
pour déboguer, mais ce ne sont que du **texte** : le daemon ne peut pas vraiment
"exploiter" l'information (compter, filtrer, déclencher une alerte...).

## La solution : un ring buffer

Un **ring buffer** (tampon circulaire) est un type de **map eBPF** spécialisé
dans l'envoi rapide d'**événements structurés** du noyau vers l'espace
utilisateur.

```
     NOYAU (eBPF)                          USERSPACE (daemon)
   +-----------------+                    +--------------------+
   | a chaque exec   |   +-----------+    | lit les evenements |
   | bloquee:        |-->| RING BUF  |--->| un par un, en      |
   |  ecrit un       |   | (tampon   |    | structure exploitable|
   |  Event{pid,comm}|   | circulaire)|   | (affiche, compte...)|
   +-----------------+   +-----------+    +--------------------+
```

"Circulaire" = quand on arrive au bout du tampon, on repart au début (comme une
piste de course). Avantages :

- très **rapide** (pas de copie inutile),
- transmet des **structures** (notre `BlockEvent` avec pid + nom + decision),
- c'est la méthode **standard moderne** (depuis le noyau 5.8) pour ce besoin,
  preferee a l'ancien "perf buffer".

## Ce qu'on transmet

On reutilise la structure `BlockEvent` (deja definie dans `shield-common`) :

```rust
pub struct BlockEvent {
    pub pid: u32,        // numero du processus
    pub comm: [u8; 16],  // nom du processus appelant
    pub blocked: u8,     // 1 = bloque, 0 = aurait bloque (log-only)
}
```

Le programme eBPF remplit cette structure et la pousse dans le ring buffer.
Le daemon la lit et l'affiche proprement, par exemple :

```
[BLOCKED]   pid=12345 comm=weirdname
[would-block] pid=12346 comm=suspicious
```

---

# 3. Ce qui est code en v0.3

1. **`shield-ebpf`** : ajout d'une map de type `RingBuf`. A chaque decision de
   blocage, on remplit un `BlockEvent` et on le pousse dans le ring buffer (en
   plus de la decision autorise/bloque renvoyee au noyau).

2. **`shield`** (daemon) :
   - lit `shield.toml` (mode + allowlist),
   - remplit les maps,
   - lit le ring buffer en boucle et affiche les evenements structures.

3. Un fichier d'exemple **`shield.toml`** est ajoute au depot.

# 4. Pour tester (rappel)

```bash
./build.sh

# avec la config par defaut (log-only)
sudo RUST_LOG=info ./target/release/shield

# pour changer la politique : editer shield.toml
```

# Glossaire rapide

- **ring buffer** : map eBPF en tampon circulaire pour evenements structures.
- **perf buffer** : ancienne methode equivalente (qu'on remplace).
- **TOML** : format de fichier de configuration simple et lisible.
- **separation code/config** : ne pas melanger la logique et les reglages.
