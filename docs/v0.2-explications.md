# v0.2 : passer de l'observation au blocage

La v0.1 observait seulement (elle loggait chaque lancement de programme).
La v0.2 ajoute la capacité de **bloquer** un programme qui n'est pas autorisé.

Le concept central de cette version, ce sont les **maps eBPF**.

---

# 1. Le concept clé : les maps eBPF

## Le problème

Le programme eBPF vit **dans le noyau** (Ring 0). Le daemon vit **en espace
utilisateur** (Ring 3). Ils ne partagent pas la même mémoire. Alors comment le
daemon dit-il au programme eBPF « voici la liste des binaires autorisés » ?

## La réponse : une map

Une **map** est une **structure de données partagée**, gérée par le noyau, que
**les deux côtés peuvent lire et écrire**. C'est le pont de communication entre
l'espace utilisateur et le noyau.

Il existe plusieurs types de maps. Pour la v0.2 on utilise une **HashMap**
(table de hachage = des paires clé → valeur, comme un dictionnaire).

```
   DAEMON (userspace)                      eBPF (noyau)
   +--------------------+                  +--------------------------+
   | ecrit dans la map: |   +----------+   | a chaque exec:           |
   |  hash("ls")   -> 1 |-->| ALLOWLIST |<--|  calcule le hash du nom  |
   |  hash("cat")  -> 1 |   | (HashMap) |   |  cherche dans la map     |
   |                    |   +----------+   |  trouve ? -> autorise    |
   |                    |                  |  absent ? -> BLOQUE      |
   +--------------------+                  +--------------------------+
```

## Pourquoi un hash (un nombre) et pas le nom en clair ?

Dans le noyau, manipuler des chaînes de caractères de longueur variable est
compliqué, et le **vérificateur** (vu en v0.1) n'aime pas ça : il ne peut pas
prouver facilement que les accès mémoire sont sûrs.

On stocke donc un **nombre** comme clé : le *hash* du nom du programme. Un hash,
c'est une fonction qui transforme un texte en un nombre de taille fixe. Avantages :
taille fixe, comparaison rapide, le vérificateur est content.

---

# 2. Choix technique : on identifie par le `comm`

Pour savoir QUEL programme se lance, deux options :

- **Le chemin complet** (ex : `/usr/bin/ls`) : le plus précis, mais difficile et
  risqué à récupérer proprement dans le hook `bprm_check_security` (beaucoup de
  code, le vérificateur peut refuser).
- **Le `comm`** (le nom court, max 16 caractères, ex : `ls`, `bash`) : simple,
  fiable, fourni directement par le noyau.

Pour une v0.2 qui marche à coup sûr, on choisit le **`comm`**.

**Limite assumée** : on ne distingue pas deux binaires différents portant le même
nom. C'est une limitation connue, qu'on documentera honnêtement, et qu'on pourra
lever en v0.3 (passage au chemin complet).

---

# 3. La sécurité avant tout : les deux modes

Un LSM qui bloque `exec` peut **rendre la machine inutilisable** si la liste est
mal remplie (imagine bloquer `/bin/bash`...). Donc on prévoit **deux modes** :

- **`log-only` (par défaut)** : on calcule la décision, on logge « j'aurais bloqué
  X », mais on **autorise quand même**. Zéro risque pour la machine.
- **`enforce`** : on bloque réellement (le programme eBPF renvoie `-EPERM`,
  c'est-à-dire « permission refusée »).

On démarre **toujours** en `log-only`. On ne passe en `enforce` que volontairement,
une fois sûr de l'allowlist.

Le mode est choisi via une variable d'environnement au lancement du daemon
(ex : `SHIELD_MODE=enforce`). Par défaut, sans rien préciser, c'est `log-only`.

---

# 4. Ce qui est codé en v0.2

1. **`shield-common`** : le type partagé de la map
   - clé = `u64` (le hash du nom du programme)
   - valeur = `u8` (un simple marqueur "autorisé")

2. **`shield-ebpf`** (noyau) :
   - déclare la map `ALLOWLIST` (HashMap)
   - à chaque exec : récupère le `comm`, calcule son hash, le cherche dans la map
   - décide : présent = autorise ; absent = bloque (mais seulement si mode enforce)

3. **`shield`** (daemon) :
   - remplit la map avec une allowlist de départ (les binaires courants pour ne
     pas se bloquer)
   - lit `SHIELD_MODE` pour choisir log-only ou enforce
   - affiche les événements

## Comment on communique le mode au noyau

Le mode (log-only / enforce) est aussi une info que le noyau doit connaître. On
le passe via une **deuxième petite map** (un tableau d'une seule case contenant
0 ou 1). Le daemon y écrit le mode au démarrage, le programme eBPF le lit.

---

# 5. Pour tester (rappel)

```bash
# build
./build.sh

# mode observation/log-only (sans danger) :
sudo RUST_LOG=info ./target/release/shield

# mode enforce (bloque reellement) — a tester avec prudence :
sudo RUST_LOG=info SHIELD_MODE=enforce ./target/release/shield
```

En `enforce`, si tu lances un programme dont le nom n'est pas dans l'allowlist,
il sera refusé avec une erreur "Permission denied".

---

# Glossaire rapide

- **map eBPF** : structure de données partagée entre noyau et userspace.
- **HashMap** : map de type clé → valeur.
- **hash** : nombre de taille fixe calculé à partir d'un texte.
- **comm** : nom court d'un processus dans Linux (max 16 caractères).
- **EPERM** : code d'erreur Unix signifiant « opération non permise ».
- **enforce / log-only** : bloquer réellement / seulement journaliser.
